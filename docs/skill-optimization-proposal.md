# Skill 功能优化方案

> 基于 HappyCapy.ai 竞品分析的优化建议
> 
> 日期：2026-04-26

---

## 竞品分析总结

### HappyCapy.ai 的 Skill 调用方案

**核心定位**：将 skill 定位为"下一代软件"——用几 KB 的文本文件替代传统几百 MB 的安装包，完全开源免费，可自由编辑。

**调用方式**：
1. **自然语言触发（推荐）**：用户直接描述需求，AI 自动识别并调用对应 skill
   ```
   用户输入：Generate an illustration of a capybara
   系统行为：自动匹配图像生成 skill 并执行
   ```

2. **手动 `/` 命令**：用户输入 `/` 触发 skill 选择器，手动浏览并选择

**Skill 发现路径**：
- 直接问 AI：`Help me find a data-processing Skill`
- Skill Store：按分类浏览（Featured / Productivity / Design / Content Creation / Development）
- 社区平台：GitHub、SkillsMP、SkillsSH

**UI 设计模式**：
- 顶部 banner 推荐"最热 skill"
- 分类横向排列，每类展示 9 个 + "See All"
- 每个 skill 卡片：标题 + 一句话描述 + 安装按钮
- 搜索入口显示总数量（325,757），强化生态规模感

**核心优势**：
- 自然语言触发，零学习成本
- Skill Store 生态规模（30 万+），形成网络效应护城河
- 开源免费降低贡献门槛

**核心弱点**：
- Skill 调用完全依赖 AI 自动识别，准确率是黑盒
- `/` 命令方式仍需用户知道 skill 名称
- 无法在未登录状态下体验任何实际功能

## 优化建议

### 一、Skill 调用入口优化

**当前问题**：用户可能需要先"知道有哪些 skill"才能调用

**优化方案**：

1. **对话框内自然语言触发**
   - 用户直接说"帮我生成小红书笔记"，AI 自动识别并调用 `xiaohongshu` skill
   - 无需用户知道 skill 名称或命令格式
   - 降低使用门槛，提升转化率

2. **保留 `/` 命令作为备选**
   - 当 AI 识别不准确时，用户可以 `/xiaohongshu` 显式调用
   - 为高级用户提供精确控制能力

3. **输入框提示优化**
   - Placeholder 文案改为："描述你的需求，或输入 / 浏览技能"
   - 引导用户了解两种调用方式

**预期效果**：
- 新用户无需学习即可使用 skill
- 高级用户保留精确控制能力
- 降低 30-50% 的使用门槛

---

### 二、Skill 发现机制

**当前问题**：用户不知道系统有哪些 skill 可用

**优化方案**：

#### 2.1 首次使用引导
- 新用户进入时，展示"热门 skill 推荐"卡片（3-5 个）
- 每个卡片包含：
  - Skill 名称 + 图标
  - 一句话描述
  - 示例用法（"试试说：..."）
  - "立即体验"按钮
- 点击后自动填充示例到输入框

#### 2.2 AI 主动推荐
- 当用户描述需求时，如果匹配到 skill，在回复中提示：
  ```
  💡 我可以用 [小红书笔记生成] 技能帮你完成，要试试吗？
  ```
- 用户点击确认后执行
- 降低用户的决策成本

#### 2.3 Skill 目录页
- 独立的 `/skills` 路由
- 按分类展示所有可用 skill：
  - 内容创作
  - 数据分析
  - 开发工具
  - 设计辅助
  - 其他
- 每个分类显示 skill 数量
- 支持展开/收起

#### 2.4 搜索功能
- 在 Skill 目录页增加搜索框
- 支持按以下维度搜索：
  - Skill 名称
  - 描述关键词
  - 标签
  - 使用场景
- 实时搜索结果展示

**预期效果**：
- 用户可以快速了解系统能力边界
- 提升 skill 使用率 50%+
- 降低"不知道能做什么"的困惑

---

### 三、Skill 调用反馈优化

**当前问题**：用户不清楚 skill 是否被调用、执行到哪一步

**优化方案**：

#### 3.1 调用确认提示
```
🔧 正在使用 [小红书笔记生成] 处理...
```
- 让用户明确知道 AI 正在调用 skill
- 增强可控感和信任感

#### 3.2 执行进度可视化
对于长时间运行的 skill（如视频生成、数据分析）：
```
🔧 [视频生成] 执行中...
  ✓ 1/4 下载素材
  ✓ 2/4 生成脚本
  ⏳ 3/4 渲染视频 (预计 2 分钟)
  ⏸ 4/4 导出文件
```
- 显示当前步骤和预计时间
- 降低用户焦虑感

#### 3.3 失败友好提示
当 skill 调用失败时：
```
❌ [小红书笔记生成] 执行失败
原因：API 配额已用完
建议：
  1. 等待 1 小时后重试
  2. 或使用 [手动创作] 功能
  3. 或升级到 Pro 版本
```
- 给出明确原因
- 提供替代方案
- 引导用户下一步行动

**预期效果**：
- 提升用户对系统的信任感
- 降低因"不知道发生了什么"导致的流失
- 失败场景下提供明确指引

### 四、Skill 元数据完善

**当前问题**：skill 的触发条件、能力边界不够清晰

**优化方案**：

#### 4.1 Skill 元数据字段扩展
为每个 skill 增加以下字段：

```yaml
name: xiaohongshu
display_name: 小红书笔记生成
description: 根据主题生成小红书风格的图文笔记
category: 内容创作
tags: [小红书, 种草, 图文, 社交媒体]

# 新增字段
trigger_keywords:
  - 小红书
  - 种草
  - 笔记
  - 红书
  - XHS

examples:
  - "帮我写一篇关于咖啡的小红书笔记"
  - "生成一个护肤品种草文案"
  - "创作小红书旅游攻略"

limitations:
  - 不支持视频笔记生成
  - 单次最多生成 1000 字
  - 需要提供明确的主题

parameters:
  - name: topic
    type: string
    required: true
    description: 笔记主题
  - name: style
    type: enum
    options: [种草, 测评, 攻略, 日常]
    default: 种草
    description: 笔记风格

usage_count: 1234
rating: 4.8
last_updated: 2026-04-20
```

#### 4.2 Skill 详情页
点击 skill 名称，展示完整信息：
- 名称 + 图标
- 详细描述
- 使用示例（可点击复制）
- 参数说明
- 能力边界
- 使用统计（使用次数、评分）
- 更新日志

#### 4.3 使用统计展示
在 skill 卡片上显示：
```
⭐ 4.8 分 | 已被使用 1,234 次
```
- 增强信任感
- 帮助用户判断 skill 质量

**预期效果**：
- 用户清楚知道每个 skill 能做什么、不能做什么
- AI 识别准确率提升（基于 trigger_keywords）
- 降低因"期望不匹配"导致的差评

### 五、Skill 生态建设

**当前问题**：skill 数量可能不足以形成网络效应

**优化方案**：

#### 5.1 突出 skill 数量
- 在首页/导航栏显示："已支持 XX 个技能"
- 强化生态规模感
- 定期更新数字，营造"持续增长"的感觉

#### 5.2 社区贡献入口
提供两种贡献方式：
1. **需求提交**：
   - "我需要一个 XXX 功能的 skill"
   - 收集用户需求，优先开发高频需求
2. **Skill 贡献**：
   - 提供 GitHub 仓库链接
   - 提供 skill 开发文档
   - 审核通过后上架

#### 5.3 Skill 排行榜
按以下维度排序：
- 最热门（使用次数）
- 最新上架
- 评分最高
- 本周新增

帮助用户发现高质量 skill

#### 5.4 Skill 组合推荐
当用户使用某个 skill 后，推荐：
```
💡 使用 [小红书笔记生成] 的用户还用了：
  - [图片生成]
  - [SEO 优化]
  - [内容排期]
```
- 提升 skill 发现性
- 增加用户粘性

**预期效果**：
- 形成"skill 越多 → 用户越多 → 贡献者越多 → skill 越多"的飞轮
- 建立社区驱动的生态系统
- 降低单一团队的开发压力

### 六、与 HappyCapy 的差异化定位

**避免正面竞争的策略**：

#### 6.1 垂直场景深耕
- HappyCapy：通用平台（30 万+ skill）
- 我们：聚焦某个垂直领域（如"中文内容创作"）
- 策略：做 10-20 个精品 skill，每个都打磨到极致
- 优势：专业度 > 广度，在细分领域建立口碑

#### 6.2 本地优先
- HappyCapy：云端沙盒，所有数据在云端处理
- 我们：本地执行，数据不出本机
- 策略：强调"隐私保护"、"离线可用"
- 优势：吸引对数据安全敏感的用户（企业、创作者）

#### 6.3 工作流编排
- HappyCapy：单次 skill 调用
- 我们：支持 skill 链式调用
- 示例：
  ```
  抓取网页 → 提取内容 → 生成小红书笔记 → 生成配图 → 发布
  ```
- 优势：解决复杂场景，提升效率 10 倍

#### 6.4 中文生态优先
- HappyCapy：英文为主，中文支持有限
- 我们：中文原生，深度适配中文场景
- 示例：
  - 小红书、微博、公众号等中文平台
  - 中文 SEO、中文文案优化
  - 中文数据分析（A 股、中文舆情）
- 优势：在中文市场建立护城河

**预期效果**：
- 避免与 HappyCapy 正面竞争
- 在细分领域建立差异化优势
- 吸引特定用户群体（中文创作者、隐私敏感用户、企业用户）

## 关键成功因素

### 1. 自然语言识别准确率
- 核心挑战：如何让 AI 准确识别用户意图并匹配 skill
- 解决方案：
  - 完善 skill 的 `trigger_keywords`
  - 收集用户真实输入，持续优化匹配算法
  - 提供"识别错误"反馈入口，快速迭代

### 2. Skill 质量 > 数量
- 避免"skill 多但不好用"的陷阱
- 策略：
  - 前期聚焦 10-20 个高频场景
  - 每个 skill 都要有清晰的使用文档和示例
  - 建立 skill 审核机制（社区贡献需审核）

### 3. 用户教育成本
- 挑战：如何让用户快速理解"什么是 skill"
- 策略：
  - 首次使用时，通过引导卡片展示 3 个示例
  - 在对话中自然引入 skill（"我可以用 XXX 技能帮你..."）
  - 提供视频教程或交互式教程

### 4. 差异化定位
- 避免与 HappyCapy 正面竞争
- 策略：
  - 聚焦中文内容创作场景
  - 强调本地执行 + 隐私保护
  - 支持工作流编排（复杂场景）

## 风险与应对

### 风险 1：AI 识别不准确
**影响**：用户说"生成小红书笔记"，AI 调用了错误的 skill

**应对**：
- 在调用前显示确认提示："我将使用 [小红书笔记生成] 技能，是否继续？"
- 提供"不是这个"按钮，让用户纠正
- 收集纠正数据，持续优化匹配算法

### 风险 2：Skill 数量不足
**影响**：用户需求无法被满足，导致流失

**应对**：
- 前期聚焦高频场景（小红书、数据分析、代码生成）
- 提供"需求提交"入口，快速响应用户需求
- 开放社区贡献，加速 skill 增长

### 风险 3：用户不理解 skill 概念
**影响**：用户不知道"什么是 skill"、"为什么要用 skill"

**应对**：
- 弱化"skill"概念，强化"能力"概念
- 文案改为："我可以帮你生成小红书笔记"（而不是"我可以调用小红书 skill"）
- 在用户使用后，再逐步引入 skill 概念

## 总结

### 核心思路
**降低认知门槛（自然语言触发）+ 提升发现性（目录/搜索/推荐）+ 增强反馈感（执行状态可视化）**

### 与 HappyCapy 的对比

| 维度 | HappyCapy | 我们的优化方向 |
|------|-----------|----------------|
| 调用方式 | 自然语言 + `/` 命令 | 同样支持，但优化识别准确率 |
| Skill 数量 | 30 万+（广度） | 10-20 个精品（深度） |
| 发现机制 | Skill Store + 搜索 | 同样支持 + AI 主动推荐 |
| 执行反馈 | 黑盒（不透明） | 透明化（进度可视化） |
| 差异化 | 通用平台 | 垂直场景（中文内容创作）+ 本地优先 + 工作流编排 |

### 预期效果
- **短期**（1-2 个月）：Skill 使用率提升 50%+，用户留存率提升 20%+
- **中期**（3-6 个月）：形成"中文内容创作"领域的口碑，吸引 1000+ 活跃用户
- **长期**（6-12 个月）：建立社区驱动的 skill 生态，skill 数量增长到 50+，形成网络效应
